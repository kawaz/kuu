## DR-099 起草 — tty は型である (`85bec49a`, 20:20:39)

DR-098 (`a396d21b`, 19:00:43 で kuu.mbt 側実装完遂) が固まった直後、kawaz
が同モデルをより単純な形へ裁定した: 「tty が何かに付く必要はない。
`{type: "tty", name: "tty"}` — セルに最初から isatty が入ってるだけの
存在で良い。微調整は tty_ 系 config で」(2026-07-12)。

DR-098 は tty 判定値を「bool 値要素に付与する wire 属性 + 値源ラダーの
新設席 (`cli > link > env > config > inherit > tty > default`)」として
モデル化していたが、この裁定により「値空間の土台そのものを差し替える
preset 型」へ転換する。DR-076 (bool は通常の値型、flag/count が特殊)
が切り開いた「特殊性は型/preset 側の展開で吸収し、値型本体を歪めない」
という筋を tty にも適用する帰結で、DR-098 が抱えていた definition-error
3 分類 (非 bool 型・値なし要素・flag/count プリセットへの `tty:` 付与)・
multiple×tty の未規定は、この転換で構文的に発生しなくなった。

要点:

- `type: "tty"` (builtin/tty) は flag/count と同族の糖衣プリセット。
  素の bool と完全に同一に振る舞い、差し替わるのは値源ラダーの
  `default` 席の解決規則だけ (`fold(観測) ?? 宣言 default ?? absent`)
- 値源ラダーは DR-031 の元の 5 段 (`cli > env > config > inherit >
  default`) に戻る。tty という独立席は消え、型ごとの `default` 解決
  ロジックとして吸収される。source タグ `tty` (観測由来 vs 宣言
  default 由来の診断) は維持
- configurable factory の config: `tty_stream` (`stdin`/`stdout`/
  `stderr`) は必須 — 未指定は definition-error。要素名からの推測は
  しない (kawaz 裁定 TTY-Q2=(c)、2026-07-12。DR-046 の name 3 軸分離の
  精神に反するため)。`tty_cygwin` は既定 true (kawaz/die DR-0008 の
  「Cygwin/MSYS pty は TTY 扱い」裁定を継承)
- `tty_provider` のシグネチャを `(stream) → bool | null` から
  `(stream) → {terminal, cygwin} | null` に改訂。fold 計算
  (`terminal || (tty_cygwin && cygwin)`) を spec 側の純データ計算として
  保つため — bool 単一値のままだと fold が provider 実装側に隠れ、
  `tty_cygwin` という config ダイヤルが実質テスト不能になる

spec 側は起草直後に立て続けに追随: `ee4a75cd` (20:20:40, DESIGN §11.4/
§12b/§13/§13.9/§16 + wire.schema/CONFORMANCE の書き換え) →
`653784017def` (20:20:40, fixtures/corpus を preset 型形式で作り直し
— `tty-ladder.json` 再構成、`tty-non-bool.json` 廃止、
`tty-stream-missing.json` 新設)。

kuu.mbt 側は同時間帯に実装完遂: `4584e43dcbc1` (20:20:46, DR-098
実装の撤去 + preset 型への転換 + fold/default 席解決 + 観測 2 値
provider) → `00c0e41e7885` (20:20:46, pin bump `fd712426`→`65378401`)。
DR-098 の台帳ロックステップ事故 (前回 journal 参照) の教訓通り、今回は

### 副産物: TypeShadow carry gap の拡大

DR-099 実装直後、kuu.mbt 側で issue が起票された
(`fba4d8ff3829`, 20:21:51): `dec_or_leaf` / `dec_positional_group`
(or 分岐・positional group 内側 leaf の decode 経路) は `TypeShadow` から
`base` と `int_round` の 2 フィールドしか carry していない。configurable
factory で作った型を or 枝や group 内側に置くと、その枝だけ config が
黙って canonical default に落ちる (エラーは出ない)。

もともと `allow_base_prefix` / `bool_config` (yes/no 方言) の 2 config が
未配線として把握されていたところに、DR-099 実装 (`is_tty` /
`tty_stream` / `tty_cygwin`) も同じ経路で未配線のまま乗った。DR-099
実装 worker は「既存の狭いスコープに合わせた (要求も fixture も無し)」
と明示的に据え置いたとの報告があり、未配線 config は 2 種類 → 4 種類
(フィールド単位では計 4 フィールド相当) に拡大した。受け入れ条件は
「spec 側に or 枝 × 方言型の fixture を先行追加 → `TypeShadow` 全
フィールド carry への一般化」の順。

## DR-100 起草 — config 語彙の整理 (`48507ef9`, 22:45:16)

DR-096 の軸別再編 (`long_`/`short_` prefix) で option 表面 config は
軸ごとの prefix 規律が確定した一方、env 軸の `auto_env` だけ軸 prefix
が後置のまま未整合だった。加えて factory config キー (DR-061) は
`number_*`/`int_*` が factory 名 prefix 付きなのに `bool_parser` の
config キー 3 つ (`true_values`/`false_values`/`case_insensitive`) だけ
bare のままという非対称もあった。kawaz 裁定 (2026-07-12) でこの 2 点を
一括整理:

- `auto_env` → `env_auto` (env_prefix と並ぶ軸 prefix 先頭の規律に統一)
- `bool_parser` の config キー 3 つに `bool_` prefix を追加
  (`bool_true_values`/`bool_false_values`/`bool_case_insensitive`)。
  意味論・default 値は不変

この過程で統括から「`bool_` prefix を外して他の factory も含め bare に
統一する」案が出たが、kawaz が「正準のリネームでこの種の綴り好みを
解決しない、alias 機構で好きにできる形が筋、canonical は prefix あり」
と裁定して撤回。canonical は「factory 名由来の prefix を持つ」で確定し、
綴りの好みを吸収する側は `docs/issue/2026-07-12-vocab-alias-installer.md`
(`57b9f70f`, 22:45:16 で起票) の vocab_alias installer 構想
(`Map<alias, canonical>` 一個で wire 語彙に糖衣 alias を足す、DR-042
不変則③ の所有語彙排他が alias-既存語彙衝突を definition-error として
自然検出する見立て) に委ねる形になった。同 issue には直後
(`270c2c72`, 22:45:17) で kawaz から設計ノートが追記され、「vocab_alias
は適用順に依存し冪等にできない — 配列先頭に置くのは定義作者の責任、
先頭に置かなければ未展開 alias が通常の unknown-vocab エラーとして
自然に表面化する」という運用が固まった。

もう一件、dd 専用キー `match`/`self` (DR-090) への `dd_` prefix 付与も
検討されたが見送り。kawaz 見解: 「dd は内部語彙っぽすぎる。定義を書く
人からするとどうでもいい知識で、prefix にする価値がない。良い代名も
浮かばないので現状維持」。

spec 側追随: `129a53b8` (22:45:16, DESIGN/schema を新語彙へ)、
`1cf38244` (22:45:17, fixtures の bool config キー追随)。
