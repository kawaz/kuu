# 2026-07-13 DR-101 サイクル (unknown_filter の definition-error 化)

filter chain 3 属性 (`value_filters`/`piece_filters`/`cell_filters` の
非 accum 位置) に指定した filter 名が registry 未登録のとき、それまで
kuu.mbt の `apply_filter_chain` が実行時に `unknown_filter` reason を
返していた挙動を、`parse_definition` の静的検査 (definition-error、
kind=`unknown-vocab`) に置き換えるサイクル。spec 起案 → spec fixture pin
→ kuu.mbt 実装 → pin bump → ロックステップ push → CI green → post-land
独立レビューまで 1 サイクルで完結し、レビューで判定マトリクスの対称性穴
1 件を発見して次サイクル送りの issue に切り出した。

## spec 起案

`d2c7a44` (`docs(decisions): DR-101 filter 名の未登録は definition-error
(unknown-vocab)`)。DR-042 (installer / filter registry の同格原則) と
DR-054 §1 (「lowering が構成できない定義 = Error」) の適用範囲を filter
側に確定させる裁定。骨子:

- filter 名の未登録は **専用 kind を新設せず既存 `unknown-vocab` に吸収**
  (DR-082 §2 の「語彙自体が未知」受け皿がそのまま該当。DR-077 §2 の
  update transform 名検査が同族の先例)
- 対象は非 accum 位置のみ。**accum 位置の cell_filters** (ARRAY filter
  registry Acc→Acc) に scalar filter registry (T→T) の綴りを書くケース
  は `invalid-range` の層違い判定 (DR-082 §2) が担う別領域
- **判定順** (DR-101 §3): accum 位置の cell_filters で「ARRAY にも
  scalar にも登録されていない綴り」は一次判定として `unknown-vocab`
  を先に適用し、`invalid-range` は「別 registry には載っているが本席の
  要求シグネチャに合わない」二次判定に取っておく
- DESIGN §8.3 (filter 名未登録の definition-error 化) / §8.5 (ref 継承
  chain 中の未登録名も判定入力に含む) に反映

spec fixture pin: `5cb4ac8` (`fixtures(definition-error): filter 名の
未登録 pin (unknown-vocab、DR-101)`) — `value-filters-unknown-vocab.json`
(scalar 要素の `value_filters`)、`piece-filters-unknown-vocab.json`
(string 域の `piece_filters`)、`cell-filters-unknown-vocab.json` (非
accum 位置の `cell_filters`) の 3 件。いずれも `nonexistent_filter` と
いう bare 綴り (`builtin` ns 糖衣、DR-094 §9) が builtin filters registry
の `owns` 集合に無いため definition-error(`unknown-vocab`) を pin する
形で、argv は書かない (DR-082 §1、definition の静的検査であり実行しない)。

## kuu.mbt 実装

`e5401c3e7d7a` (`fix(core): DR-101 filter 名の未登録は parse_definition
の静的検査で reject (unknown-vocab)`)。`installer.mbt` に
`collect_unknown_filter` を新設し、value_filters/piece_filters/cell_filters
(非 accum 位置) の filter 名を filters registry (scalar T→T,
`lookup_filter`) の owns 集合と照合、未登録なら `DUnknownVocab` を
definition-error として収集する完全性検査に組み込んだ。

accum 位置の判定は DR-101 §3 の判定順に合わせて更新: 既存の
`collect_invalid_accum_cell_filter` を「scalar registry に IN かつ
ARRAY registry に NOT IN」の場合だけを二次判定 (`invalid-range`) として
拾うよう絞り込み、両 registry どちらにも無い綴りは
`collect_unknown_filter` 側の一次判定 (`unknown-vocab`) に委ねた
(両検査の判定条件が交錯すると同じ入力が二重に reject される、または
判定順が逆転する事故を避けるための絞り込み)。

`apply_filter_chain` (filters.mbt) の runtime `unknown_filter` Err
分岐は、definition-time gate 通過後は到達不能になるため no-op
pass-through に変更 (既存 `apply_cell_filter_chain` と同型の defensive
posture)。`filters_wbtest.mbt` の runtime `unknown_filter` reason
wbtest は削除 — spec 側の 3 fixture が pin を引き受ける形に一本化した。

pin bump: `8c91113ddf9b` (`ci: spec fixtures pin bump d390df20 ->
5cb4ac8`)。

conformance (kuu.mbt 側 `just check` + spec runner): decoded=209 /
ran_cases=553 / skipped=0 / mismatches=0、moon test 323 件 pass で
green を確認済み。

## Worker 訂正 2 点

実装過程で worker が自ら発見・修正した判断転換 2 点。いずれも実物 grep
による訂正で、委譲時点の誤認が実装前に解消された。

- **filter chain 属性名の DR-034 統合形へ**: 委譲時に使われた属性名
  (`filters`/`pre_split_filters`/`post_filters`/`rate_of_change`/
  `on_value_selected`) は DR-009 (filter chain の3段) 当時の名残で、
  DR-009 自体は DR-034 (multiple structure) により旧 filter chain
  三分割宣言として superseded 済み。実物は `value_filters`/
  `piece_filters`/`cell_filters` の統合形に置き換わっている。worker が
  `installer.mbt`/`DESIGN.md` の実物 grep で発見し、DR-101 本文・
  fixture・実装のいずれも統合後の 3 属性名で揃えた
- **判定順の DR-101 §3 整合**: 当初案のまま `collect_invalid_accum_cell_filter`
  の判定条件を広く取ると、accum 位置で「両 registry に無い綴り」も
  invalid-range 側に拾われてしまい DR-101 §3 が規定する「一次
  unknown-vocab → 二次 invalid-range」の判定順と矛盾する。worker が
  判定条件を「scalar IN & ARRAY NOT IN」に絞り込むことで、両 registry
  判定の交錯を回避し §3 の判定順どおりに実装した

## Signing SHA 変化の対処

spec push は signing を経由するため、push 前のローカル SHA と push 後に
origin へ載る SHA が異なる (同一内容・同一 commit message でハッシュ
だけ変わる、post-DR-100 journal サイクルの `c8a1de91`→`19a187e0`
[post-DR-100 journal 記載] と同型の事象)。DR-101 fixture pin commit も
push 前ローカル SHA `227eee16` が push 後 origin では `5cb4ac8` に変化
した。kuu.mbt 側の pin bump commit を spec push 前のローカル SHA
(`227eee16`) 参照で先に作っていたため、spec push 完了後に origin の
確定 SHA (`5cb4ac8`) へ pin ref を差し替える修正が必要になった。最終的
に `8c91113ddf9b` は正しく `5cb4ac8` を指す形で確定している。

## Post-land review

DR-101 land 直後、節目レビューとして codex 経由のレビューを試みたが
環境障害に遭遇した:

- codex CLI (0.144.1) を subscription 認証で起動すると
  `gpt-5.3-codex is not supported when using Codex with a ChatGPT
  account` エラーで拒否される
- `gpt-5.6-sol` (CLIProxyAPI 経由) へのモデル切替で codex exec も
  試みたが、`code-mode host exited during handshake` で cell/Read tool
  が機能せずレビュー続行不能

codex 経路が両方とも塞がっていたため、代替として opus47-worker-high
(dr101-review) による独立レビューに切り替えた。VERDICT: **PASS + High
1 件**。

High 1 件: DR-101 §3 の判定マトリクスが 6 セル中 5 セルしか規定して
いない対称性欠陥。抜けているのは「非 accum 位置 × cell_filters ×
ARRAY-only 綴り (例: `unique`)」の帰結で、現行実装
(`installer.mbt:3684-3690`) は scalar registry の owns 集合のみで判定
するため unknown-vocab で reject するが、accum 位置の対称論理 (「別
registry には載っているが本席の要求シグネチャに合わない = invalid-range」)
を非 accum 側にも適用するなら invalid-range が本来の座席という指摘。
独立 issue `4bb26362`
(`docs/issue/2026-07-13-dr101-non-accum-array-only-filter-vocab.md`、
status: idea) として起票し、今サイクルは PASS のまま land・CI green を
確定させ、対称性穴は次サイクルの裁定に送った。
