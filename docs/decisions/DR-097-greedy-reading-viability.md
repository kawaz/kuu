# DR-097: 先食い・早閉じ抑制の「読める」の精密化 — 成立した読みへの選好

> 由来: DR-096 の bare long (`long_prefix: ""` 無条件合法化) 固定を経て、先食い規定 (DR-041 §4) の述語「読める」がトリガ一致を指すのか読みの成立を指すのかが `height height` (両トークンとも値照合失敗) のようなケースで問われた。kawaz 裁定 (2026-07-12) による確定。

## 決定

DR-041 §4 の先食い (および同型の早閉じ抑制) が使う述語「読める」を精密化する:

> 先食い (および同型の早閉じ抑制) の述語「読める」は**トリガ一致ではなく、読み自身の成立**を指す: トリガ発火に加え、その entry 自身の値スロット消費が argv から確保でき (missing operand でない)、値空間照合 (parse 相、factory config 込み — DESIGN §3.3 / DR-061 §5 の「相」線引き) を通ること。**下流の帰結 (同スコープの他要素の充足、遅延述語 §15.9、後続トークンの成否) は判定に参加しない。** 成立する読みが 1 本も無い消費点は読みゼロと同じ扱いになり素通し枝が生きる。そこから完全経路が立てば採用され、greedy 側の held Error は捨てられる (DR-037「他ルートが通るならエラーは捨てる」の適用)。

具体的には、ある消費点で greedy エントリのトリガがトークンに一致しても:

- その読み自身が成立する (値スロットを持たない flag / count / dd / command は常に成立。値持ちは値トークンを確保でき、かつ値空間照合を通る) なら、先食いは維持される (素通し枝は生成されない、従来どおり)。**下流で経路が死ぬ場合 (制約未充足・同スコープの他要素の欠落等) も維持され**、その失敗は held Error 付きの failure として可視のまま残る
- 成立する読みが 1 本も無い (値トークン枯渇 = missing operand / 値空間照合失敗) なら、その消費点は読みゼロと同じ扱いになり、素通し (positional が raw で食う読み) の枝が生成される
- **生成と裁定の層分離**: 素通し枝の**生成**は parse 相の情報だけで決まる。生成された全枝の**裁定** — 完全経路の勘定 (DR-038)、制約による経路選別 (DR-047 §6-1)、held Error の取捨 (DR-037) — は従来の教義がそのまま司る

### 論証: 既存教義の精密化であって新規則ではない

1. 先食いの目的 (DR-041 §4、垂直スライス PoC 根拠) は、greedy 読みと素通しの**両方**が完全経路になりうる場面での恒常 ambiguous を防ぐこと (anti-ambiguity の選好)。greedy 側の読みが1本も完全経路に到達しない場合、この目的は働かない — 抑制しても防ぐべき ambiguity は存在せず、唯一残りうる解釈 (素通し側) を黙って潰すだけになる。
2. DR-037 は「解けた (Accept 完了した) 枝の数」で結末を決め、経緯節で「途中エラーは評価を遅延して全ルートチェックして他が通るならエラーは捨てられる」ことを明文で確定している。トリガ一致という粗い述語で他ルート (素通し) の**生成自体**を封じる読み方はこの教義と整合しない。本 DR が精密化するのは生成条件のみ (parse 相の述語に揃える) で、生成後の裁定層には手を入れない — 全ルートチェックは従来どおり働く。
3. DR-038 は「第2の完全経路を黙って潰すのは契約違反」とする (`-n1.0f` の greedy commit 非採用の論拠)。同型の違反が逆方向にも起こりうる: 成立しない読みのために唯一の完全経路 (素通し側) を黙って潰すのも同じ契約違反である。

4. (実測根拠) 述語を「完全経路 (遅延述語込み) の存在」まで広げる別定式化は、参照実装 (kuu.mbt) での実装実験で棄却した — 無関係な下流失敗が先食い判定に逆流して既存 fixture 39 件が破壊される (詳細は「採用しなかった案」)。

これらから、「読める」の精密化は DR-037/038/041 の既存教義から演繹される帰結であり、新しい選好規則の追加ではない。

## 帰結 (fixture 固定)

kawaz 裁定 (2026-07-12) による確認例。定義: `config: {long_prefix: ""}`、options: `height` (number, long: true)、positionals: `rest` (string, repeat min 0)。

1. `height height` — 両トークンとも height の読みが成立しない (値照合失敗 / missing operand) → 両方素通し → `{rest: ["height", "height"]}` success
2. `height 168.5 height` — token1-2 は成立 (先食い維持) → `{height: 168.5}`。token3 は続く operand がなく不成立 → 素通し → `{height: 168.5, rest: ["height"]}` success
3. `height height 168.5` — token1 は値 "height" が number 照合失敗で不成立 → 素通しで rest へ。token2 は "168.5" で成立 (先食い) → `{height: 168.5, rest: ["height"]}` success

fixture: `fixtures/matcher-readings/long-empty-prefix-typed.json`。

## 不変な既存挙動

- `prog -v src` (`-v` は flag、値スロットを持たないため読みが常に成立=消費対象トークンなしで即座に完結) → src="-v" の読みは立たないまま。定義済み option と同形のリテラルを positional に渡す escape は従来どおり dd (`--`) が要る (DR-041 §4 の帰結は不変 — 変わるのは「読みが成立しない」場合だけ)
- dd (`--` は素の exact 一致で失敗しうる値照合を持たないため常に成立) の抑制は不変
- 受け皿 positional の無い定義では従来どおり failure + held Error 表示 (`fixtures/matcher-readings/cluster-split-no-flag.json` の `-n1.0f` 等)。精密化が効くのは素通し枝の**生成先** (受け皿) がある場合のみで、受け皿が無ければ held Error が唯一の結末になる点は変わらない — 親切なエラーは受け皿がある時だけ受容と引き換えに消える、という DR-041 受け入れる帰結 #1 (typo が positional に飲まれうる) と同じ向き
- `fixtures/path-search/held-errors-distinct-depth.json` / `held-errors-same-depth.json` / `fixtures/matcher-readings/prefix-guard-number.json` / `prefix-guard-string.json` は greedy vs 素通しの競合 (先食い規則の対象) を含まない (トリガ自体が不一致の未定義トークン、または or 枝競合) ため不変
- **制約 (遅延述語) の未充足は先食い・早閉じに影響しない**: `height` (number, long, requires: weight) に `height 168.5` を渡して weight が欠けている場合、読み自身は成立しているため先食いは維持され、従来どおり requires_violated の可視 failure になる (無言で positional に格下げされない)。制約の充足は値源不問 (DR-093 — env / config 経由でも充足しうる) ため、これを判定に入れると環境変数の有無で同じ argv のトークンが option になったり文字列になったりする層逆転が起きる — 判定を parse 相で閉じることがこれを構造的に排除する

## 早閉じ抑制への適用

DR-041 §4 は早閉じ抑制を「先食いの適用先を『素通し消費』から『スコープ退出』にも広げた同型規則」と定義しており、述語「読める」を共有する。したがって本 DR の精密化はそのまま及ぶ: スコープ内トリガがトークンに一致しても、その読みを経由する完全経路が成立しないなら、スコープ退出は妨げられない。スコープ退出側は先食いと失敗の質が異なる (どのスコープがトークンを読むか = どのコマンドの引数になるか自体が変わる) ため、専用 fixture `fixtures/command-scope/early-close-viability.json` で固定する: 読みが成立する間はスコープを閉じられない (従来どおり) / 成立する読みが無ければスコープを閉じて親の背骨に返す / 制約未充足は退出判定に参加しない (可視 failure のまま)。

## typo マスクの受容拡大

`height 16O.5` (typo、number 照合失敗) は、寛容な positional 受け皿がある定義では黙って rest に吸われ success になる — 現行の held Error 表示より受容側に倒れる。これは DR-041 §5 の prefix ガード非採用が既に受け入れた帰結 #1 (typo が positional に飲まれうる) の一貫適用であり、新しいトレードオフではない。緩和策も同じ: (a) 静的 warn (DR-021「option 群 + 上限なし string positional」構造の警告)、(b) 近接候補提示 (findings F-016)、(c) 方言ガードの opt-in。コアの既定は定義を信頼する。

## 停止性と有界性

述語は当該読みの範囲 (トリガ + 自身の値スロット、高々数トークン) で閉じ、下流の探索を再帰的に参照しないため自明に停止する。bounded path-search (§15.2 / DR-038) の有界性も不変 — 判定は読み生成時の局所検査で済み、探索の呼び出し構造を変えない。

## 採用しなかった案

### トリガ一致のままの現状維持

DR-041 §4 の字面 (トリガ一致 = 読める) を変えない案。値照合が failure する消費点でも先食いが素通し枝を封じ続けるため、唯一残りうる解釈を構造的に破壊する。DR-038 の「経路を黙って潰さない」契約に違反し、DR-096 の bare long (`long_prefix: ""`) 固定によって顕在化した `height height` のようなケースで唯一の妥当な読み (素通し) が失われる。

### 完全経路 (遅延述語込み) の存在で判定する

「読める」を §15.1 の完全解決経路 (遅延述語 §15.9 込み) の存在で判定する案。DR-037 の全ルートチェックに字面上最も忠実で、「下流失敗で greedy 経路が死ぬなら素通しが唯一の解釈になる」ケースも救えるが、参照実装での実装実験 (2026-07-12) により棄却した:

- **無関係な下流失敗の逆流**: 判定が経路全体を見るため、当該読みと無関係な失敗が読みを「不成立」に変える。最小反例 = `fixtures/dd/basic.json::missing-positional` (`-a -b` + 構造的必須 positional x): flag `-b` の読み自体は健全なのに x の欠落が経路全体を殺すため、`-b` が素通し解放されて `x="-b"` の無意味な success に化け、本来の missing_operand failure が消える — 「値スロットを持たない flag は常に成立」という不変挙動の主張とこの定式化自体が両立しない
- **制約の層逆転**: requires 等の充足は値源不問 (DR-093) のため、env / config の値の有無で同じ argv のトークンが option になったり文字列になったりする。「requires の相手を付け忘れただけ」の正しい綴りの入力が、親切な requires_violated ではなく無言の positional 丸呑み success に化ける
- **repeat 取り分選好 (DR-043) との文脈依存**: 判定を取り分選好の特定文脈 (repeat を続ける側) の中で行うと「打ち切って先へ進めば完全経路がある」場合を見落とし、well-founded な定式化・実装が複雑化する。実装実験では既存 fixture 39 件が flip し、局所的な修正では収束しなかった
- 判定が下流探索を再帰参照するため、停止性 (残余トークン長の帰納法) と有界性の実装ノートを別途要する
