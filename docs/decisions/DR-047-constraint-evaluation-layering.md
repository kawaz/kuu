# DR-047: 制約評価のレイヤリング — 遅延述語は完全経路の成立条件、required は値充足

> 由来: findings `2026-06-29-ast-missing-pieces.md` の F-004 (空 argv と required) / F-026 (requires の評価タイミング) / F-027 (exclusive_group の評価タイミング) と、旧実装考古学 (findings `2026-07-03-core-archaeology.md` テーマ 2 — 旧研究の即時/遅延バリデータ分類と投機実行スケッチ)。本セッションの議論で確定。**DR-016 / DR-031 の「required は committed を見る」を覆す**。

## 決定

### 1. 制約評価は 2 レイヤ (即時 / 遅延)

- **即時述語**: トークン単体で判定可能な検証 (型パース、regex、in_range 等)。既存の filter (DR-037 の Reject/Error、DR-034 の pieceProcessor) がこの層であり、本 DR では変更しない
- **遅延述語**: 経路全体の最終状態を見ないと判定できない検証 (`required` / `requires` / `exclusive_group`)。本 DR の対象

### 2. 遅延述語は完全経路の成立条件 (経路フィルタ)

DR-038 の「完全**解決**経路」の成立条件に遅延述語を含める。制約を満たさない経路は完全経路と数えない (0 / 1 / 2+ の計数に入らない)。**1 本確定後の事後検証層は持たない**。

- 先食い・早閉じ抑制 (DR-041 §4)・shadowing (DR-042 global) と同じメタ原理に属する: 経路間の優先規則を評価器に持ち込まず、**経路の成立条件 (読める・食える・解決できる) の精密化で 1 本に絞る**。「そのスコープで食えるなら可能な限り消費する」の制約版
- 旧実装考古学の投機実行スケッチ (「遅延バリデータは全フォーク世界の全引数消化が終わった時点で評価する」) の直系。旧モデルは名前完全一致でフォークが発生せず post-parse チェックに縮退していたが、現行の複数読み (DR-041 §3) がフォークを供給するため、この席が初めて意味を持つ

### 3. 述語失敗は Error (DR-037 への接木、新語彙なし)

DR-037 の「解けた枝 = filter Error 含めて全段が通った」の「全段」に遅延述語を含める。遅延述語違反は **Error** (この経路のつもりだが制約違反 → 経路不成立、エラーは保持し、完全経路 0 本時に表示する)。Reject (静か脱落) にはしない — 「required X missing」は失敗時の表示価値が高い。

### 4. 評価対象は値源ラダー適用後の最終状態

遅延述語は、経路の消費が終わり値源ラダー (DR-031: env / config / inherit / default) の充填を済ませた**最終状態**に対して評価する。空 argv (F-004) の特別扱いは無い: 消費ゼロの経路が値源充填後に遅延述語を通れば成功、required 未充足なら完全経路 0 本で失敗 (保持された Error を表示)。

### 5. 判定入力は制約の意味論から決まる (required は値充足)

| 制約 | 述語の種類 | 判定入力 |
|---|---|---|
| `required` | 値述語 (「アプリは値を必要とする」) | 最終状態の値の有無 (default 込み) |
| `exclusive_group` | 指定述語 (「同時に指定するな」) | committed (同グループで最大 1 つ) |
| `requires` | 混合 | トリガ側 = committed (「明示した時」)、目的語側 = 値の有無 (「相手の値が要る」) |

- **required は値充足**: `{"name": "paymentmethod", "or": [...], "required": true, "default": "cache"}` は未指定でも充足する — アプリが必要とする値は default が供給している。required の実質は「default 等の値源を持たない要素への明示強制」+「結果に必ず値がある」という型保証 (言語バインディングが `T?` でなく `T` を生成できる。F-022 と接続)。**DR-016 / DR-031 の「required は committed/selected を見る、default で埋まっただけでは満たさない」を覆す**
- **exclusive_group / requires トリガは committed**: DR-045 の効果制御と噛み合う — unset (`--no-x`) で取り消した要素は排他の衝突にも requires のトリガにも数えない。selected (入口マッチ) は診断メタに留め、制約判定には使わない
- **committed の定義は本 DR で変更しない** (DR-045: 効果が明示制御する。env / config の充填では立たない)。requires の目的語が default の値で満たされる (例: 既定の鍵パス) のは required と同じ値述語の帰結

> **明確化 (kawaz 裁定 2026-07-09): requires の目的語が bool 型 (flag preset 含む) の場合、充足条件は「値の有無」ではなく「解決後の値が true であること」(値源不問 — cli / env / config / inherit / default のどれ経由でも true なら充足、false なら不充足)。** 根拠: bool は preset の暗黙 default:false (LOWERING §A.5) を常に持つため、「値の有無 (default 込み)」判定だと bool 目的語の requires が一度も指定されなくても vacuous に充足し、制約が黙って no-op になる (kuu.mbt 2026-07-09 実測、installer_wbtest の A/B pin)。committed 基準にしなかったのは env / config 経由の true (例: `YES=true` が `--force requires --yes` を満たす) を弾かないため。「requires x = x が真」という bool の真理値としての自然な読みとも一致する。非 bool 型は従来どおり「値の有無 (default 込み)」。実装への含意: bool 目的語の判定は値源ラダー解決後の値に依存する。制約は遅延述語なので、判定は**全完全経路の収集後に、値源ラダーが解ける後段 (resolve 層、post_filters と同じ側) で経路フィルタとして適用**する — eval 層 (経路探索) に値源を持ち込まない。ambiguous 解消への参加 (§「制約が ambiguous の解消に参加する」) は「後段フィルタで経路数が 1 に減ればその経路で確定、0 なら failure (全違反が errors に載る)」として同じ意味論のまま成立する (kawaz 整理 2026-07-09)。

> **精密化 (DR-093, kawaz 裁定 2026-07-11): required / requires の判定入力は型 (値空間) への委譲として一様定式化される。** 上表の「required = 最終状態の値の有無」「requires 目的語 = 値の有無 (bool は解決後値が true)」は**値空間を持つ要素**が前提。値空間が空の要素 (`type: "none"`、dd 含む、DR-089) は required / requires 目的語のいずれでも「発火したこと (committed)」で充足する。bool 目的語の判定 (直上の明確化) は「bool 型の充足定義」として同じ型委譲の枠に位置づけ直される。判定入力に committed を使う点は本 DR の線 (selected を診断メタに留める) と整合し、変更しない。

### 6. 受け入れる帰結

1. **制約が ambiguous の解消に参加する**: 構造の異なる複数の完全経路のうち制約を満たすものが 1 本だけなら、ambiguous ではなくその経路で静かに確定する (例: `-n1.0f` の 2 読み (DR-038) で分割読み側の `-f` が requires 未充足なら、値付着読み `n="1.0f"` で成功)。DR-043 の取り分選好が ambiguous 検出を保存する (選好は取り分次元に閉じる) のと異なり、遅延述語は経路の成立自体を変える
2. **env + cli の exclusive 非検出**: `FORMAT_JSON=1` (env) + `--yaml` (cli) は衝突にならない (env は committed を立てない)。結果には json / yaml の両値が現れうる。値レベルで 1 つに絞りたい定義は `{"name": "format", "or": [...]}` で 1 要素に束ねるのが正道 (定義を信頼する、§0.1)
3. **required + default は常に充足**: 「default があるのに明示も強制」は表現不能になる (実用ユースケースが無いと判断した)

### 7. 診断モードは lint 系譜 (本 DR では確定しない)

制約違反の経路も計数に含めて ambiguous を可視化する「事後検証型」の見え方は、canonical には採用せず、開発時診断 (DR-021 の warn / lint、DESIGN §15.6) の系譜に置く。本 DR では診断モードの具体仕様を確定しない。遅延述語が経路列挙から独立した純述語である構造 (適用位置を差し替え可能な形) は実装上保つ。

## 採用しなかった案

### 事後検証型 (1 本確定後の制約チェック) を canonical にする

旧実装の post-parse チェックと同型。path-search がトークン消費だけで閉じるレイヤ分離の綺麗さはあるが、片方が制約違反と分かっている 2 経路を ambiguous と報告することになり、「そのスコープで食えるなら可能な限り消費する」の一貫原理に反する。診断としての価値は §7 (lint 系譜) へ。

### 制約評価タイミングの利用者向けスイッチ (プラガブル化)

両型の差は「述語適用が計数の前か後か」の 1 点でありスイッチ可能な構造だが、観測差が出るのは複数読みフォークを持つ kuu 固有の状況のみで、既存 CLI 互換の要求からこのスイッチが要求される場面は想定できない (既存パーサはフォーク自体を持たない)。canonical を 1 本に固定し、中核契約 (経路の数え方) の 2 モード化と conformance の倍化を避ける。

### env / config の充填で committed を立てる

required を committed 判定のまま救うための案。required を値述語にすれば不要であり、committed の純度 (パース時の明示操作、DR-045) を保つ方が一貫する。source タグ (DR-031) が由来の区別を既に保存している。

### 遅延述語失敗の第 3 語彙 (「Error 保持つき脱落」の新設)

DR-037 の Error 意味論の適用先拡張 (「全段」に遅延述語を含める) で表現できるため、新語彙は不要。

## 射程外

- 違反時のエラー報告の JSON 構造 (どのグループ / 要素が衝突したか) は本 DR で確定しない (F-043、後続 issue 起票候補。DR 番号は未割当)
- 条件付き制約 (値依存の requires) は or 構造への誘導方針 (F-028) のまま、本 DR で確定しない
- `at_least_one` 等の追加の制約語彙は、必要になった時に本 DR の 2 レイヤ + 判定入力 (値述語 / 指定述語) の枠組みで分類する
